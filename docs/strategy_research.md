# Strategy Research Harness v1

Strategy Research Harness v1 is a deterministic hypothesis-testing layer for stored
prediction-market data. It builds as-of feature snapshots, runs explicit research
strategies, creates hypothetical intent proposals, evaluates each proposal through
Pre-Trade Gate v1, optionally routes approved hypothetical intents through the Paper
Execution Simulator, and stores simulated attribution summaries.

This is not live trading. Research signals and proposals are hypotheses, not execution
instructions. The harness does not route orders, connect to venues, use credentials,
or access live accounts.

## ResearchStrategyDefinition

`ResearchStrategyDefinition` stores a research configuration: name, version, family,
default requested size, default intent type, strategy context, config, and metadata.
Definitions contain no credentials, wallet data, venue account identifiers, or live venue
fields.

Default v1 strategies are deterministic and idempotent:

- `baseline_research_only_v1`
- `trust_verdict_allow_filter_v1`
- `integrity_pass_filter_v1`
- `divergence_research_hypothesis_v1`
- `composite_conservative_research_v1`
- `scenario_context_research_v1`

## ResearchFeatureSnapshot

`ResearchFeatureSnapshot` is a generic as-of feature container. V1 can build features
from market data, resolution/trust verdicts, integrity assessments, equivalence
assessments, divergence assessments, pre-trade decisions, paper snapshots, and scenario
feature snapshots.

`feature_source=SCENARIO_SIMULATION_PLACEHOLDER` carries local, imported scenario features.
V1 does not execute MiroFish or call external services.

## ResearchSignal

`ResearchSignal` is a strategy-generated research signal. It records signal type,
strength, confidence, action bias, reason codes, source feature IDs, source reference IDs,
and deterministic input/output hashes.

Signals are labeled as research signals. They are not live venue action.

## ResearchIntentProposal

`ResearchIntentProposal` is a hypothetical proposal generated by a strategy. It must be
converted into a `TradeIntent` and evaluated by Pre-Trade Gate before any optional paper
simulation can occur.

Proposals contain no venue order IDs, account identifiers, credentials, wallet fields, or
live execution fields.

## ResearchDecisionTrace

`ResearchDecisionTrace` links the research signal/proposal to the pre-trade decision and
optional paper output. It stores pre-trade action, paper order status, simulated filled
size, simulated average fill price, and reason codes.

If pre-trade blocks a proposal, the trace is still stored, but no paper fill is created.

## ResearchRun

`ResearchRun` is a synchronous run-once batch. It scans a deterministic timestamp grid,
selected markets, and selected strategies. It records created signal, proposal, pre-trade,
paper order, paper fill, and error counts. There is no daemon or background scheduler.

## ResearchRunSummary

`ResearchRunSummary` aggregates counts by strategy, signal type, pre-trade action, paper
order status, and reason code. It also stores simulated filled size and final simulated
portfolio metrics when paper data exists. Simulated metrics are not real performance.

## ResearchAttributionReport

`ResearchAttributionReport` stores grouped attribution by strategy, market, venue, reason
code, signal type, pre-trade action, and paper order status. Simulated PnL fields are
explicitly named `simulated_pnl_by_strategy` and `simulated_pnl_by_market`.

No Sharpe, real PnL, EV, fills against venues, or live performance claims are computed.

## Default Strategies

`baseline_research_only_v1` validates plumbing. It emits a research-only signal and a
research-only hypothetical proposal.

`trust_verdict_allow_filter_v1` uses the latest as-of trust verdict. It creates proposals
only when the trust verdict action permits; review or blocked verdicts produce no
proposal.

`integrity_pass_filter_v1` uses the latest as-of integrity assessment. Severe integrity
action hints block proposals; passive-only context creates passive-style proposals.

`divergence_research_hypothesis_v1` uses latest as-of divergence assessments. It only
acts on comparable, non-review divergence context and only creates a hypothetical BUY
proposal for the lower aligned price side when that side can be determined safely.

`composite_conservative_research_v1` requires trust, quality, integrity, pre-trade, and
divergence context to pass conservative filters. Missing critical data produces a
review-only signal and no proposal.

`scenario_context_research_v1` uses scenario features only to generate `WATCH` or
`REVIEW_ONLY` signals. It never creates proposals.

DataOps coverage and gap reports are not strategy triggers in v1. They help researchers
inspect whether fixture or public-read collection has enough rule, price, liquidity, and
quality coverage before interpreting research results.

## No-Lookahead Behavior

Research feature building uses repository methods that honor as-of semantics:

- Market data uses `available_at <= asof_timestamp`.
- Integrity, equivalence, divergence, pre-trade, paper, scenario, and research list methods use
  `available_at <= asof_timestamp` where applicable.
- Backfilled historical market data remains hidden from research until its `available_at`.
- Trust and rule context use existing point-in-time repository semantics.

Generated research features set `available_at` to the research as-of timestamp so replay
can use them deterministically.

## Gate And Paper Dependencies

Proposal evaluation always calls Pre-Trade Gate before optional paper simulation.

When paper simulation is enabled, only approved hypothetical intents are sent to the Paper
Execution Simulator. Paper outputs remain clearly labeled simulated. The research harness
does not make live orders or use real account positions.

The paper simulator also enforces the pre-trade gate invariant when it receives an intent.
Research summaries count proposal-to-pre-trade outcomes at the trace level, while paper
orders retain their own pre-trade decision reference. This keeps attribution readable even
when the same deterministic pre-trade decision is reused across a proposal trace and a
simulated paper order.

## Replay

Replay steps attach research metadata when research outputs exist as of the replay
timestamp:

- latest research signal IDs
- latest research proposal IDs
- latest research trace IDs
- signal and proposal counts
- pre-trade action counts

`research_policy_v1` maps the latest stored `ResearchDecisionTrace` pre-trade action to a
replay action. It does not generate new proposals or run paper simulation inside replay.

## Later Support

The generic feature snapshot interface can later accept MiroFish-style scenario features
through `SCENARIO_SIMULATION_PLACEHOLDER`. The same trace and attribution structure can
also support strategy replay, risk attribution, market-making filters, and execution
management design without adding live venue access in this round.
